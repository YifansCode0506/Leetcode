# 枚举右维护左核心兵法库

恭喜你！刷完这一组题后，你已经跨过了算法思维里一个非常重要的分水岭。

从最初的“两数之和”“找对应负数”，到后面的“数位和”“等价骨牌”“数对和”，再到“宽高比”“完整天数”“二进制相似度”，这些题看起来外壳完全不同，但它们背后都有同一套骨架：

> **枚举右，维护左**：把当前元素当成右端点 `j`，把它左边已经出现过的信息压缩成一个“账本”，然后用当前元素去账本里找答案。

这套模型非常值钱。它不是某一道题的技巧，而是一类题的共同底层语言。

---

## 一、核心心法：为什么它能把暴力降维？

暴力双重循环通常是这样想的：

```js
for (let i = 0; i < n; i++) {
    for (let j = i + 1; j < n; j++) {
        // 判断 nums[i] 和 nums[j] 是否能配对
    }
}
```

它的问题是：每来一个新的 `j`，都要回头把所有 `i` 再扫一遍。

“枚举右，维护左”的想法是：

```js
for (let j = 0; j < n; j++) {
    // 1. 把 nums[j] 当成右端点
    // 2. 去左侧账本里查它需要的历史信息
    // 3. 用查询结果更新答案
    // 4. 把 nums[j] 登记到账本，供未来使用
}
```

它的本质是：**不再反复扫描过去，而是把过去提前整理成可以 O(1) 或接近 O(1) 查询的状态。**

这就是为什么很多 $O(n^2)$ 的题，可以被压成 $O(n)$ 或 $O(n \log n)$。

---

## 二、三条铁律：查账、登记、边界

### 1. 当前元素永远扮演“右角色”

循环走到 `nums[j]` 时，你要强迫自己这样问：

> 如果当前元素是右端点，那么它需要左边出现过什么？

比如：

* 两数之和为 `target`：当前 `x` 需要左边有 `target - x`
* 两个数相同：当前 `x` 需要左边有 `x`
* 两个余数凑满 60：当前余数 `r` 需要左边有 `(60 - r) % 60`
* 字符集合相同：当前字符串的 `mask` 需要左边出现过同样的 `mask`

一旦能说出“当前需要什么左伴侣”，这道题基本就被打开了。

### 2. 先查账，后登记

绝大多数要求 `i < j` 的题，都要严格遵守：

```js
const c = cnt.get(key) ?? 0; // 先查左边历史
ans += c;                   // 再用历史更新答案
cnt.set(key, c + 1);         // 最后登记当前元素
```

这个顺序非常关键。因为当前元素还没登记时，账本里天然只包含它左边的元素，所以不需要额外写 `if (i < j)`。

如果先登记再查询，就容易把 `i = j` 也算进去。

### 3. 左侧账本不一定是 Map

“维护左”里的账本，不只是哈希表。它可以是任何能代表左边历史的压缩状态：

* 一个数字：左侧最小价格 `minPrice`
* 两个数字：左侧所有数组的最小开头 `mn` 和最大结尾 `mx`
* 固定数组：余数计数、数位和计数、最高位计数
* `Map`：大范围数字、字符串 key、比例 key
* `Set`：窗口内是否存在某个数
* 数组里的数组：每种余数对应的所有历史下标
* 一个前缀和变量：前面所有水平线的组合数总和

所以不要把这个模型理解窄了。它真正维护的是“左边对当前有用的信息”，至于用什么容器，要看题目。

## 三、账本类型：题目不同，记法不同

| 账本类型 | 典型题目 | 账本里记什么 | 核心动作 |
| --- | --- | --- | --- |
| **单值最优状态** | 买卖股票、最大递增差值 | 左边最小值 | `ans = Math.max(ans, x - min)` |
| **左右极值状态** | 数组间最大距离 | 历史数组的最小首元素、最大尾元素 | 当前数组和历史极值交叉计算 |
| **存在性登记 (Set)** | 存在重复元素 II | 窗口内有没有某个数 | `set.has(x)` 命中即返回 |
| **最新位置覆盖** | 最小连续卡牌数、重复元素距离 | 某个值最近一次出现的下标 | `ans = Math.min(ans, j - last.get(x) + 1)` |
| **频次增量累加** | 好数对、等价骨牌、互换矩形、相似字符串 | 某个特征出现过几次 | `ans += count`，再 `count++` |
| **可消耗库存** | K 和数对、数对和 LCCI | 左边还没被配走的数字数量 | 找到互补数就配对并扣库存 |
| **固定范围计数** | 被 60 整除的歌曲、完整天数、美丽数对 | 余数、最高位、数位和等小范围状态 | 用数组代替 Map，速度更快 |
| **状态压缩** | 相似字符串对 | 字符集合压成二进制 `mask` | 用一个整数代表一个集合 |
| **真实轨迹恢复** | 完整天数拓展版 | 每种状态对应的所有历史下标 | 当前 `j` 和所有历史 `i` 连线 |
| **组合前缀累加** | 梯形计数 | 前面所有水平线的两点组合数总和 | 当前线组合数 `k` 与历史总和 `s` 相乘 |

这张表的重点不是背题，而是看懂每一题到底在“记什么”。

## 四、几个最容易混淆的分支

### 1. `ans += count` 和“消耗库存”不是一回事

有些题是统计所有配对数量，比如好数对、等价骨牌、互换矩形：

```js
const c = cnt.get(key) ?? 0;
ans += c;
cnt.set(key, c + 1);
```

当前元素可以和左边所有同类元素分别组成一对，所以直接加 `c`。

但 `Max Number of K-Sum Pairs` 和 `Pairs With Sum LCCI` 是“配完就不能再用”的库存模型：

```js
const need = k - x;
const c = cnt.get(need) ?? 0;

if (c > 0) {
    ans++;
    cnt.set(need, c - 1); // 互补数被消耗
} else {
    cnt.set(x, (cnt.get(x) ?? 0) + 1);
}
```

这里不能 `ans += c`，因为当前 `x` 只能和一个历史元素配一次。

### 2. “最新位置”为什么要覆盖旧下标？

求最小距离、最短区间时，只关心离当前最近的同值元素。

所以每次遇到 `x`，都应该覆盖：

```js
if (last.has(x)) {
    ans = Math.min(ans, j - last.get(x) + 1);
}
last.set(x, j);
```

旧下标更远，不可能帮当前元素得到更短答案。

### 3. 滑动窗口题要删除过期历史

`Contains Duplicate II` 的限制是 `abs(i - j) <= k`，所以左侧账本不能无限增长。Set 版本的关键是：

```js
if (set.has(nums[i])) return true;
set.add(nums[i]);
if (i >= k) set.delete(nums[i - k]);
```

这个题不是单纯“维护所有左边”，而是维护“距离当前不超过 `k` 的左边”。

### 4. 比例、数组、对象不要直接当 Map key

JS 的 `Map` 用对象或数组当 key 时，比的是引用地址，不是内容。

```js
new Map().set([1, 2], 1).get([1, 2]); // undefined
```

所以“可互换矩形”不能直接拿 `[w / g, h / g]` 当 key。更稳的写法是化成字符串：

```js
const g = gcd(w, h);
const key = `${w / g}/${h / g}`;
```

这样 `"3/2"` 和另一个 `"3/2"` 会被认为是同一个 key。

### 5. 浮点数比例有精度风险

宽高比这类题，不建议直接写：

```js
const key = w / h;
```

JavaScript 的数字是双精度浮点数，某些除法结果可能有精度误差。最稳的方式是用最大公约数把比例化成最简整数比，再拼字符串。

## 五、JS 实战细节：写得稳一点

### 1. 数字计数优先用 `??`，少用 `||`

`||` 判断的是“真假”，会把 `0` 当成没有值：

```js
const c = map.get(x) || 0;
```

在很多计数题里，`0` 是一个合法值。更稳的写法是：

```js
const c = map.get(x) ?? 0;
```

`??` 只在左边是 `null` 或 `undefined` 时才使用默认值。

一句话记忆：**账本里存数字，默认优先用 `??`。**

### 2. 重复读取 Map 时，先缓存

下面这种写法会多次查同一个 key：

```js
if (map.has(target) && map.get(target) > 0) {
    map.set(target, map.get(target) - 1);
}
```

更舒服的写法是：

```js
const c = map.get(target) ?? 0;
if (c > 0) {
    map.set(target, c - 1);
}
```

好处有两个：逻辑更清楚，也避免重复查表。

### 3. 固定小范围时，数组比 Map 更直接

如果 key 的范围很小，比如：

* 余数：`0 ~ 23`、`0 ~ 59`
* 最高位：`1 ~ 9`
* 骨牌特征：`11 ~ 99`
* 数位和：范围有限

直接用数组通常更清晰：

```js
const cnt = Array(60).fill(0);
ans += cnt[(60 - t % 60) % 60];
cnt[t % 60]++;
```

Map 适合 key 范围很大、很散、或者 key 是字符串的情况。

## 六、四步破题公式

以后遇到新题，可以按这四步走。

### 第一步：定关系

先问：当前 `x = nums[j]` 需要左边有什么？

常见关系：

* 和为 `k`：需要 `k - x`
* 差为 `k`：需要 `x - k` 或 `x + k`
* 特征相等：需要同样的 `key`
* 余数互补：需要 `(mod - r) % mod`
* 求最大利润：需要历史最小值
* 求最小距离：需要同值的最近下标

### 第二步：定特征

不是所有题都直接拿原值当 key。很多题要先提炼特征：

* 等价骨牌：`min(a, b) * 10 + max(a, b)`
* 互换矩形：`${w / g}/${h / g}`
* 相似字符串：字符集合 `mask`
* 美丽数对：历史数字的最高位、当前数字的最低位
* 梯形计数：每条水平线上的点数 `c`，再转成组合数 `c * (c - 1) / 2`

题目的难点经常不在 Map，而在“到底应该把什么放进 Map”。

### 第三步：定账本

根据 key 的范围和答案需求选容器：

* 只需要最值：用变量
* key 范围小：用固定数组
* key 范围大：用 `Map`
* 只问存在性：用 `Set`
* 需要下标：存 index
* 需要输出所有配对：存 index 数组
* 有距离限制：记得删除过期元素

### 第四步：定更新顺序

大多数 `i < j` 题：

```js
// 1. 计算当前 key / target
// 2. 查左侧账本
// 3. 更新答案
// 4. 登记当前元素
```

但如果题目先要整体分组，比如梯形计数，也可以先统计每组数量，再在“组”这个层面枚举右、维护左。

## 七、这组题的统一模板

### 频次计数模板

```js
let ans = 0;
const cnt = new Map();

for (const x of nums) {
    const key = getKey(x);
    const c = cnt.get(key) ?? 0;
    ans += c;
    cnt.set(key, c + 1);
}

return ans;
```

适合：好数对、等价骨牌、互换矩形、相似字符串。

### 互补目标模板

```js
let ans = 0;
const cnt = new Map();

for (const x of nums) {
    const target = getTarget(x);
    ans += cnt.get(target) ?? 0;

    const key = getKey(x);
    cnt.set(key, (cnt.get(key) ?? 0) + 1);
}

return ans;
```

适合：歌曲总时长能被 60 整除、完整天数、两数关系类计数。

### 最新位置模板

```js
let ans = Infinity;
const last = new Map();

for (let j = 0; j < nums.length; j++) {
    const x = nums[j];
    if (last.has(x)) {
        ans = Math.min(ans, j - last.get(x));
    }
    last.set(x, j);
}

return ans === Infinity ? -1 : ans;
```

适合：最小距离、最短连续区间。

### 可消耗库存模板

```js
let ans = 0;
const cnt = new Map();

for (const x of nums) {
    const target = k - x;
    const c = cnt.get(target) ?? 0;

    if (c > 0) {
        ans++;
        cnt.set(target, c - 1);
    } else {
        cnt.set(x, (cnt.get(x) ?? 0) + 1);
    }
}

return ans;
```

适合：每个元素最多使用一次的配对题。

## 八、最后的判断信号

看到下面这些字眼，要立刻想一想能不能套“枚举右，维护左”：

* 找两个元素满足某种关系
* 统计满足条件的数对数量
* 要求 `i < j`
* 求最近一次、最小距离、最短区间
* 当前元素只需要知道左边的某种压缩信息
* 暴力做法是双重循环，但内层查询可以被提前整理成账本

真正熟练以后，你会发现很多哈希、滑动窗口、前缀状态、甚至部分动态规划题，都是这套模型的变形。

**一句话收束：枚举右，是确定当前问题；维护左，是把过去变成答案可以直接调用的资料库。**
